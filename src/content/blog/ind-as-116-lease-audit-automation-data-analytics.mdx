## Ind AS 116 Lease Audit Automation with Data Analytics

If you are signing off on a statutory audit for a listed entity with 600 active leases, manually reconciling every right-of-use asset and lease liability schedule is simply not sustainable. The question is no longer whether to use data analytics for Ind AS 116 testing — it is how to build a workflow that your team can actually execute within a compressed audit window. This article walks you through exactly that.

---

### Why Ind AS 116 Creates a Reconciliation Burden for Auditors

Ind AS 116 requires an auditor to verify that every single lease contract has been captured, measured, and unwound correctly through its full term. Right-of-use asset and lease liability schedules demand line-by-line recalculation across every contract — opening balance, interest accrual, repayment, depreciation, and closing balance — for each reporting period. Miss one input, and the error compounds forward.

The problem scales badly. Modifications, reassessments, and variable lease payments introduce non-linear breaks in the amortisation schedule that cannot be spotted by simply casting a column total. A mid-year rent revision, an exercised renewal option, or an index-linked escalation all require the liability to be remeasured at a revised discount rate from the modification date — and that remeasurement ripples through every subsequent period.

For listed companies and large unlisted entities filing under Ind AS, carrying 500 or more active leases is routine. Retail chains, telecom operators, logistics companies, and hospital groups all fall into this category. Traditional tick-and-tie procedures — picking a sample, tracing numbers by hand — simply cannot achieve meaningful coverage across a population that size without blowing the audit budget entirely.

---

### Core Data Inputs Required Before Any Analytics Can Begin

Before you open any analytics tool, you need two clean, complete data sets from the client. The first is the lease register — a structured file listing every active lease with the following fields at minimum: commencement date, original and remaining lease term, incremental borrowing rate (IBR), renewal option terms, variable payment clauses, and functional currency. If the client's register is missing IBR documentation or has blank fields for renewal options, raise this immediately as a data quality finding — building analytics on incomplete inputs will produce unreliable exceptions.

The second set comprises ERP exports: the GL trial balance for all Ind AS 116 account codes (ROU asset, accumulated depreciation, lease liability — current and non-current, interest expense, and lease payments made), plus the depreciation run file and interest accrual journals. Request these exports at the period-end trial balance date, and ensure they carry full account codes and cost-centre tags so you can split by segment later.

Supporting documents complete the picture. You need signed lease agreements for the top 20 leases by liability value, modification memos approved by management, and the board-approved IBR basis document or a certificate from the client's bankers or valuer confirming the discount rate. These are not optional — SA 500 requires sufficient appropriate evidence, and a client-produced spreadsheet without an underlying IBR justification is not sufficient on its own.

### Step-by-Step Data Analytics Workflow for Lease Audit Procedures

The workflow below covers the full cycle from raw data ingestion through to a prioritised exceptions list. Depending on your toolset, execution time for a 300-lease portfolio should fall between four and eight hours of analyst time — compared with several weeks of manual work.

1. Import the client-provided lease register into your analytics environment. If you are using Excel Power Query, load the CSV or Excel extract directly and lock the source file as read-only before any transformation. Python users should read the file into a pandas DataFrame and immediately write a hash of the raw file for your audit trail.
2. Cleanse and standardise the data. Convert all date fields to a consistent ISO format (YYYY-MM-DD), align currency figures to INR using period-end exchange rates for foreign-currency leases, and flag any records with null values in commencement date, IBR, or lease term as exceptions requiring manual follow-up before proceeding.
3. Map GL account codes from the ERP export to the corresponding lease register line items. This cross-reference table is critical — it tells you which journal entries belong to which contract and prevents double-counting when the client has split a single lease across multiple cost centres.
4. Rebuild the independent amortisation schedule for every lease using auditor-controlled IBR inputs. Calculate opening lease liability, interest charge for the period (opening balance × IBR × days/365), lease payment made, and closing liability. Run this calculation month by month if the client uses monthly schedules, or annually if the portfolio uses calendar-year unwinding.
5. Compute the ROU asset schedule independently: opening net book value, depreciation charge (straight-line over the shorter of lease term and useful economic life), any impairment, and closing net book value. Cross-cast to ensure the depreciation charge agrees to the client's depreciation run file at the account-code level.
6. Compare your auditor-computed closing lease liability and ROU net book value for each contract against the corresponding figures in the client's own schedule. Compute the variance in absolute terms and as a percentage of the client figure. This is your primary output file.
7. Apply materiality thresholds. Set a lower threshold (say, 10% variance or INR 5 lakh, whichever is lower) for individually significant leases and a higher threshold for the tail of low-value contracts. Contracts breaching the threshold are flagged as exceptions for further investigation.
8. Sort exceptions by variance magnitude and assign a risk rating — high (possible IBR error or missed modification), medium (rounding or date convention difference), and low (below individually significant threshold but above trivial). This prioritisation tells your team where to spend substantive testing hours.
9. Generate a timestamped exception report listing each flagged contract, the auditor figure, the client figure, the variance, and the assigned risk rating. Save the transformation scripts and all parameter inputs (IBR table, materiality threshold, exchange rates) as permanent audit workpaper attachments. This is your evidence file for the engagement partner's review.

### Manual vs Automated Lease Audit Procedures: A Feature Comparison

| Dimension | Manual Tick-and-Tie | Data Analytics Approach |
| --- | --- | --- |
| Population coverage | Typically 20–30% sample of leases based on value or random selection | 100% population testing across all active contracts |
| Time per lease | 25–40 minutes per contract for recalculation and cross-referencing | Batch processing — full portfolio in minutes once model is built |
| Handling of modifications | Each modification traced individually; high risk of one being missed | Date-based filters isolate all modification events automatically |
| Audit documentation quality | Handwritten workpapers or manual Excel files prone to transcription errors | Reproducible, timestamped scripts and exception logs |
| Scalability across audit years | Effort repeats in full each year | Prior-year model rolls forward with updated inputs; incremental effort only |
| SA 520 analytical procedure compliance | Relies on auditor judgement in selecting comparatives | Systematic variance against independently computed benchmark |

The time difference alone justifies the investment. On a 400-lease portfolio, manual procedures at 30 minutes per lease consume 200 person-hours. The analytics approach cuts that to under 15 hours of build-and-review time once the model is established.

### Handling Lease Modifications and Reassessments in the Analytics Model

Your client is a listed retail chain with 350 store leases. During the year, 40 leases were modified — 25 due to COVID-era rent concessions still being unwound, and 15 due to fresh renegotiations at lease renewal. The client's lease software has updated the schedules, but your job is to independently verify the remeasurement was done correctly at each modification date.

Start by pulling all GL journals posted to the lease liability account between 1 April and 31 March. Filter for entries with an unusual debit-credit pattern — a direct debit to lease liability without a corresponding lease payment credit typically indicates a remeasurement gain, while an unexplained credit indicates a modification that increased the liability. Cross-reference these journal dates against the modification memos you collected at the planning stage.

For each of the 40 modified leases, recalculate the revised lease liability at the modification date. Use the remaining lease payments from that date forward, discounted at the revised IBR applicable on the modification date (not the original commencement date IBR). If the client has used the original commencement IBR for a modification that constitutes a separate lease under Ind AS 116 paragraph 44, that is a misstatement — quantify it and report it to the engagement partner immediately.

Flag any leases subject to index-linked rent reviews as a separate high-risk category. These require the auditor to independently obtain the relevant index (typically CPI or WPI) at the reassessment date and verify that the client has remeasured the liability using updated lease payments reflecting the revised index. Many clients omit this reassessment entirely, treating index escalations as variable lease payments instead — which is incorrect under Ind AS 116 paragraphs 42 and 45.

### Disclosure Adequacy Testing Using Automated Data Extraction

Ind AS 116 paragraph 58 requires specific disclosures in the notes — a maturity analysis of undiscounted lease liabilities, total cash outflows for leases, short-term lease expense, low-value lease expense, and variable lease payments not included in the measurement of the liability. Manually reconciling these disclosures to the underlying schedules is tedious and frequently skipped or performed only at a high level. Analytics makes this testable in minutes.

Once your independent amortisation schedule is complete, use a pivot summary to bucket future undiscounted lease payments by maturity band: within one year, between one and five years, and beyond five years. Compare these totals directly to the maturity analysis disclosed in Note X of the financial statements. Variances here are almost always disclosure misstatements, because the underlying schedules are typically correct but the note preparer used a different cut-off date or excluded certain lease categories.

Cross-check the total cash outflows for the year — this is the sum of principal repayments and interest payments as they appear in the cash flow statement — against your GL extract for lease liability debit entries and finance cost entries coded to Ind AS 116. For short-term and low-value leases, pull the relevant GL codes and verify the expense disclosed in the notes matches the sub-ledger total. These two checks together cover the most commonly misstated Ind AS 116 disclosures in Indian financial statements.

Finally, use pivot summaries disaggregated by business segment or geography to verify that ROU asset disclosures in the segment note agree to the sum of individual lease schedules assigned to each segment. This is particularly relevant for conglomerates and companies with multiple reportable segments under Ind AS 108, where the segment-wise split is often prepared separately from the consolidated lease schedule and can diverge.

### Practical Implementation Guide for Indian CA Firms

Tool selection should match your firm's current data capability — do not over-engineer the solution in Year 1.

1. Assess your firm's current analytics maturity. If your team is comfortable with Excel and basic formulas but has no Python or IDEA experience, start with Excel Power Query combined with structured VBA macros for the amortisation loop. This handles portfolios up to roughly 200 leases without performance issues and requires no new software licences.
2. For mid-tier practices handling five to ten listed clients per partner, IDEA or ACL (now Galvanize) is the right investment. These tools offer built-in audit trail functionality, which directly satisfies the documentation requirements under SA 500 and SA 520 without additional workpaper effort. Budget approximately INR 2–3 lakh per year per licence and factor this into your fee proposals.
3. If your firm already has staff with Python experience — even one or two analysts — build the amortisation schedule engine in pandas. A well-written Python script handles 1,000+ leases in under two minutes, produces reproducible output, and integrates with your existing document management system through file export. Document all scripts with inline comments and version-control them using Git so the engagement partner can review the logic independently.
4. Design your workpaper structure before the pilot engagement begins. The analytics file, transformation scripts, IBR input table, materiality worksheet, and exception report must each be saved as separate, clearly labelled attachments in your audit file. ICAI's SA 230 requires audit documentation to be sufficient for an experienced auditor to understand the procedures performed — a single output spreadsheet with no parameter log does not meet this standard.
5. Run a pilot on one listed client with a straightforward lease portfolio — ideally a company with fewer than 150 leases, no foreign-currency contracts, and no mid-year modifications. Use this engagement to refine your model, identify data quality issues that are specific to your clients' ERP systems (SAP versus Oracle versus Tally-based solutions produce very different export formats), and build a reusable template.
6. After the pilot, document lessons learned in a firm-level methodology note. Update your audit programme templates to reference the analytics procedure as the primary substantive test for Ind AS 116, with manual follow-up reserved for high-risk exception items only. This is the point at which the time saving becomes structural rather than one-off.
