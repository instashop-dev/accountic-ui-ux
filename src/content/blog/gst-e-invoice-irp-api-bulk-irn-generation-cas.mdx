## GST E-Invoice IRP API: Bulk IRN Generation for CAs

If you are managing e-invoicing for more than five or six clients, the manual IRP portal is already costing you time and creating reconciliation risk. API-based IRN generation is no longer a luxury for large firms — it is the only reliable approach at scale. Here is everything you need to implement it properly.

---

### Why CAs Need IRP API Automation in 2024

The e-invoicing threshold dropped to Rs 5 crore aggregate turnover, and that single change pulled the vast majority of your B2B client base into mandatory compliance. What was once a concern for large manufacturing clients is now a weekly operational task for mid-size traders, service providers, and distributors on your roster.

The manual IRP portal breaks predictably under volume pressure. Session timeouts generate duplicate IRN attempts, partial batch uploads create reconciliation gaps, and separate login sessions per GSTIN turn a 200-invoice day into a three-hour manual exercise. None of this scales.

API automation changes the equation entirely. Sub-second batch processing, structured error responses, and full audit trails mean your team handles exceptions rather than supervising every submission. The investment in integration pays back within weeks for any practice running more than 500 invoices per month across clients.

---

### IRP API vs. Manual Portal: Head-to-Head Comparison

Before committing to integration effort, your partners and senior managers need a clear picture of what the API actually buys you versus the status quo.

| Dimension | Manual IRP Portal | IRP API Integration |
| --- | --- | --- |
| Throughput capacity | Single invoice per submission; no batch support | Hundreds of IRNs per batch call; parallelisable across GSTINs |
| Error handling | Generic failure screens with no machine-readable codes | Structured JSON error codes enabling programmatic retry logic |
| Multi-client scalability | Separate login session required per GSTIN | Multi-GSTIN credential pools managed in a single integration layer |
| Audit trail depth | Screenshot-level records only | Full cryptographic QR payload, signed invoice hash, and API timestamp |
| Session reliability | Timeouts during high-traffic periods cause silent failures | Stateless API calls with token refresh logic eliminate timeout risk |

The audit trail difference is particularly important. When your client faces a GST audit or scrutiny notice, the API response log gives you a signed, timestamped record for every IRN that no portal screenshot can replicate.

### IRP API Architecture and Authentication Essentials

There are four live IRP endpoints — NIC, Cygnet, Clear, and EY — all running identical JSON schemas but maintaining independent sandbox environments. For pre-production testing, you should run your integration against at least two sandbox environments to confirm your implementation is not accidentally dependent on endpoint-specific behaviour.

Authentication runs on a two-layer model. At the transport layer, your GSP credentials authenticate your application to the IRP. Inside that authenticated session, each taxpayer's username and password generate a session token specific to that GSTIN. This means credential management and token lifecycle are separate concerns in your architecture.

Access tokens expire every six hours without exception. If you are running overnight batch jobs across 30 GSTINs, a hard-coded token will cause silent failures at the 3 AM run without any obvious error surfacing until the next morning. Build automated token refresh logic as a first-class requirement, not an afterthought.

One step many integrations get wrong: the IRP mandates public key encryption of the taxpayer password using an RSA key published by the IRP before every authentication request. Submit a plaintext password and you will receive a cryptic authentication error that wastes debugging hours. Retrieve the current IRP public key as part of your integration setup and re-fetch it periodically since it is rotated.

### Step-by-Step Workflow: Generating Bulk IRNs via IRP API

This is the production workflow we recommend for CA practices moving from raw invoice data to validated IRNs across multiple client GSTINs.

1. Extract invoice data from the client's ERP or accounting software and normalise it into a staging database with a unique internal reference for each invoice record.
2. Map the staging data to the e-invoice JSON schema version 1.1, paying particular attention to mandatory fields like SellerDtls, BuyerDtls, ItemList, and ValDtls that cause the highest proportion of validation failures.
3. Run local schema validation against the published e-invoice JSON schema before any API call. This catches structural errors without consuming your API call quota or incurring chargeable failed hits against your GSP plan.
4. Chunk the validated invoice queue into batches of 50 invoices per API call. This batch size sits comfortably within IRP rate limits of 100 calls per minute per GSTIN and gives you manageable retry units when partial batch failures occur.
5. Authenticate against the IRP for each GSTIN in scope, retrieve the session token, and queue the batch calls with controlled concurrency so you do not breach the per-GSTIN rate limit.
6. Submit each batch, capture the full API response — including IRN, signed QR code, acknowledgement number, and acknowledgement date — and write it immediately to your IRN response store.
7. Write the IRN and signed QR code back to the source ERP or accounting system within the same transaction that marks the invoice as submitted. Never allow a state where your IRN store and the source system are out of sync.
8. Move any failed invoice records to a retry queue, classify errors by code, apply the appropriate remediation logic (schema fix versus GSTIN correction versus duplicate resolution), and reprocess in the next batch cycle.

Schema validation in Step 3 deserves emphasis. A single malformed JSON field like an incorrect HSN code length or a mismatched invoice type code will fail the entire batch in some IRP configurations. Local pre-validation is your first line of defence and your cost control mechanism.

### Handling API Errors, Cancellations, and IRN Reconciliation

**Case Scenario:**

Consider a mid-size textile wholesaler client — 300 to 400 invoices per month, two GSTINs across states, and a busy month-end close. After going live with API integration, the practice team starts seeing a 4 percent failure rate on the first production run. Two error codes account for almost all failures.

Error code 2150 — duplicate IRN — appears because the source ERP had already pushed some invoices through the manual portal the previous day during a parallel testing phase. Remediation here is not a retry: you need to query the IRP to fetch the already-generated IRN, reconcile it into your store, and mark the invoice as completed rather than failed. Blind retries will keep generating 2150 errors indefinitely.

Error code 2271 — invalid GSTIN in buyer details — appears on 12 invoices where the client's master data carried stale GSTINs for counterparties. These require human review and GSTIN verification on the taxpayer search portal before resubmission. Your retry logic must route 2271 errors to a human-review queue, not an automatic retry loop.

IRN cancellation adds another layer of operational discipline. Cancellation is only permitted within 24 hours of IRN generation, and critically, an IRN linked to an e-way bill cannot be cancelled even within that window. This means your pre-submission validation must be thorough, because post-generation options are severely limited.

Build a daily reconciliation run that cross-matches your API response log against the GSTR-1 auto-population data pulled from the GST portal. Any IRN appearing in your log but absent from auto-population is a flag worth investigating before GSTR-1 filing. Catching this gap daily costs minutes; catching it during filing costs hours and potentially triggers late filing penalties.

Permanently failed invoices — those that cannot be automatically remediated — must land in a dead-letter queue with a human-review flag. Silent compliance gaps, where an invoice was never converted to an IRN and neither the CA nor the client is aware, are the single largest risk in poorly designed e-invoice automation.

### Multi-Client IRP Integration Checklist for CA Practices

Getting your GSP selection and onboarding process right before going live saves significant rework. Use this workflow to structure the rollout.

1. Evaluate at least three GSP options — NIC direct, Cygnet, Clear, EY, or others — scoring each on SLA uptime guarantees (target 99.9 percent), sandbox environment quality, per-API-call pricing at your expected volume, and documented support response time. Choosing solely on price typically leads to poor sandbox parity and slow incident resolution.
2. Design your credential vault architecture before onboarding a single client. Each client GSTIN's API credentials must be isolated behind role-based access controls so your junior staff handling client A cannot see or use credentials for client B. Use a secrets management tool rather than a shared spreadsheet.
3. Set up your monitoring stack with three mandatory alerts: token expiry approaching within 30 minutes for any active GSTIN, batch failure rate exceeding 2 percent over any rolling 15-minute window, and IRP endpoint latency spikes above 3 seconds sustained for more than 60 seconds.
4. For each new client, verify e-invoice enablement status on the GST portal before attempting any API call. A GSTIN that is not e-invoice enabled will fail at authentication, not at invoice submission, which can be confusing to diagnose.
5. Complete a full sandbox test cycle for each client GSTIN covering at least five invoice types: B2B standard, credit note, debit note, export with payment, and export without payment. Obtain a signed test sign-off from the client before moving to production.
6. Execute a signed data-processing agreement with each client before their GSTIN credentials are loaded into your system. This is not just a good practice — it is your professional protection when handling taxpayer authentication credentials on behalf of a client.
