## Context

The project is an Astro + Cloudflare Workers blog automation platform. The admin panel (`/admin/*`) is protected by a single shared password stored as a Bearer token / `httpOnly` cookie. The audit found the password hardcoded in `src/middleware.ts` and `src/pages/admin/login.astro`, no CSRF protection on any POST route, inconsistent security headers, and a missing path-traversal guard before GitHub file commits.

The fixes are surgical: no new libraries, no architectural changes. The admin session model (cookie + middleware check) stays the same.

## Goals / Non-Goals

**Goals:**
- Remove hardcoded `ADMIN_PASSWORD` from source; read from Cloudflare Secret at runtime
- Add CSRF protection (Origin/Referer validation) to all six state-changing admin POST routes
- Centralize and consistently apply security headers across all admin API routes
- Add path-traversal guard on slug before GitHub file path construction (two callsites)
- Add input length validation on prompt save endpoint
- Add job-status guard on replay endpoint (only `failed` jobs replayable)
- Move hardcoded `GITHUB_OWNER`, `GITHUB_REPO`, `GITHUB_BRANCH` to env vars

**Non-Goals:**
- Adding MFA, session refresh tokens, or idle timeout (separate concern)
- Encrypting R2 snapshots at rest
- Rotating GITHUB_TOKEN (operational task, not code)
- Adding rate limiting to login endpoint (operational task via Cloudflare rules)
- Adding security.txt (content/infra task)
- Replacing manual SQL escaping with parameterized queries in seed-prompts.ts (script, not production code)

## Decisions

### 1. CSRF: Origin/Referer header validation instead of token-based CSRF

**Decision:** Validate that `Origin` or `Referer` header matches the site's own origin on every admin POST. No separate CSRF token stored in cookie or DOM.

**Rationale:** The admin routes are API endpoints, not form-submit targets for external pages. Origin/Referer checking is sufficient for same-site cookie protected APIs because:
- The `admin_token` cookie is `sameSite: strict`, which already blocks cross-site cookie inclusion in modern browsers
- A shared CSRF token would require an additional round-trip (GET the token, POST with token), adding complexity without meaningful security gain for this single-admin use case
- Astro does not have a built-in CSRF token mechanism; Origin/Referer is standard for API endpoints

**Alternative considered:** Double-submit CSRF cookie pattern — rejected because it requires client-side JS changes and state management across all admin pages.

**Implementation:** Create a `validateCsrf(request: Request, expectedOrigin: string): boolean` utility in `src/lib/admin-security.ts`. Call it at the top of each POST handler; return 403 on failure.

### 2. Security headers: shared constant, not middleware

**Decision:** Export an `ADMIN_SECURITY_HEADERS` object from `src/lib/admin-security.ts` and spread it into each route's response headers manually, rather than adding a global middleware layer.

**Rationale:** The existing `src/middleware.ts` only handles auth; inserting a separate header-injection middleware creates ordering concerns with Astro's middleware chain and affects non-admin routes. Since there are only ~6 admin API routes, the duplication cost is low and each route stays self-contained.

**Alternative considered:** Cloudflare Worker `fetch` handler wrapping all responses — rejected because it would require restructuring the Workers setup and could affect the public site inadvertently.

### 3. `ADMIN_PASSWORD` secret: runtime env lookup in both middleware and login

**Decision:** In `src/middleware.ts`, read `context.locals.runtime.env.ADMIN_PASSWORD`. In `src/pages/admin/login.astro`, read `Astro.locals.runtime.env.ADMIN_PASSWORD`. No default value fallback — throw if missing, to catch misconfigured deployments immediately.

**Rationale:** Silent fallback to a hardcoded default would defeat the purpose. A loud runtime error during local dev (where the secret is not set) is better than a silent security hole in production. Local dev can use a `.dev.vars` file (Cloudflare's convention for local Worker secrets).

### 4. Slug path-traversal guard: strict allowlist regex

**Decision:** Before constructing the GitHub file path, validate `slug` matches `/^[a-z0-9-]+$/`. Reject slugs containing `/`, `..`, or any character outside lowercase alphanumeric and hyphens.

**Rationale:** Slugs are generated by `toSlug()` in `src/lib/slug.ts` which already normalizes to this character set, but slugs stored in D1 from older pipeline runs (or via direct DB edit) may not have been through that function. An explicit guard at the callsite is the safest defense.

**Alternative considered:** `path.resolve()` + prefix check — rejected because Cloudflare Workers runtime does not expose Node's `path` module; the regex guard is simpler and fully portable.

### 5. GitHub repo constants: move to `env` without breaking existing behavior

**Decision:** In `src/pages/admin/api/refresh.ts`, read `GITHUB_OWNER`, `GITHUB_REPO`, and `GITHUB_BRANCH` from `context.locals.runtime.env` with the current hardcoded values as defaults (`?? 'instashop-dev'`, etc.) to avoid breaking existing deployments without a simultaneous secrets update.

**Rationale:** These are not security secrets (just metadata), so a fallback default is acceptable. This makes them overridable for staging without requiring an immediate env var rollout.

## Risks / Trade-offs

- **Local dev will fail if `ADMIN_PASSWORD` is not set in `.dev.vars`** → Document required `.dev.vars` entries in a comment near the `wrangler.jsonc`; add to README if applicable.
- **Strict `sameSite: strict` + Origin check may cause issues if Cloudflare proxies alter the Origin header** → Test against production URL; Cloudflare passes the original `Origin` header through unchanged for Workers.
- **Slug regex `/^[a-z0-9-]+$/` could reject existing slugs with uppercase or underscores** → Review slug generation in `src/lib/slug.ts` to confirm it already enforces this; if not, also patch `toSlug()`.
- **Replay guard (only `failed` jobs) may block legitimate retries of `pending` jobs** → Acceptable trade-off; an admin can change job status directly in D1 if needed.

## Migration Plan

1. **Provision the secret before deployment:** `wrangler secret put ADMIN_PASSWORD` (for both `wrangler.jsonc` and `wrangler.pipeline.jsonc` environments if needed).
2. **Add `.dev.vars` for local development:** `ADMIN_PASSWORD=<local-dev-password>` — not committed to source control.
3. **Deploy code changes** — the admin panel will be inaccessible until the secret is provisioned, so step 1 must precede step 3 in production.
4. **Rollback:** Revert the commit; re-deploy the previous version. The secret does not need to be removed — old code ignored it.
