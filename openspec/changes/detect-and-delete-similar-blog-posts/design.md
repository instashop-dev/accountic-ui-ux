## Context

The blog corpus is generated by a 6-stage Cloudflare Workers pipeline. Topic-discovery deduplicates by exact title only, so semantically similar articles accumulate over time (e.g. "How to Respond to Section 148 Notice" and "Responding to Income Tax Notice Under Section 148"). The pipeline already contains `computeBigramJaccard()` in `src/lib/regression.ts`, and all posts are plain markdown files in `src/content/blog/` with YAML frontmatter parseable via `src/lib/frontmatter.ts`. D1 holds corresponding rows in `posts`, `drafts`, and `generation_jobs`.

## Goals / Non-Goals

**Goals:**
- Provide a standalone CLI script that reads the local blog corpus, computes pairwise bigram-Jaccard similarity across post bodies, and outputs a ranked Markdown + JSON report of similar pairs above a threshold.
- Provide a deletion script (or `--delete` flag) that removes identified posts from the filesystem and their D1 rows, defaulting to dry-run.
- Reuse existing utilities (`computeBigramJaccard`, `parseFrontmatter`) — no new packages.
- Keep both scripts runnable via `npx tsx scripts/…` with `wrangler` D1 access for deletions.

**Non-Goals:**
- Real-time pipeline-level duplicate blocking (a separate future concern).
- Embedding-based or LLM semantic similarity (overkill for the current corpus size).
- Deduplication of `topics` or `drafts` tables (only published posts in scope).
- Automated selection of which post to keep (user always decides).

## Decisions

### Decision 1: Bigram-Jaccard on body text only (not title)
**Chosen**: Strip frontmatter, compute bigrams on remaining markdown body.  
**Alternative considered**: TF-IDF cosine similarity — more nuanced but requires a vector library.  
**Rationale**: Bigram-Jaccard is already in the codebase, runs in pure TypeScript with zero deps, and is battle-tested by the humanizer regression gates. For a 200–500 post corpus, O(n²) comparisons (~25K–125K) complete in seconds.

### Decision 2: Separate detect and delete scripts (not a single script with modes)
**Chosen**: `detect-similar-posts.ts` (read-only, always runs) + `delete-similar-posts.ts` (destructive, dry-run by default).  
**Alternative considered**: Single script with `--delete` flag.  
**Rationale**: Separating concerns makes the detection step safe to run in CI or on a schedule without risk of data loss. The delete step requires deliberate invocation and explicit slug confirmation.

### Decision 3: Delete from filesystem first, then D1
**Chosen**: Delete the `.md`/`.mdx` file, then `DELETE FROM posts WHERE slug = ?`, then cascade-delete dependent rows.  
**Rationale**: Filesystem is the source of truth for published content (Astro reads it at build time). D1 rows without a filesystem file are harmless but inconsistent; removing from both keeps state clean. D1 deletion order: `generation_jobs` → `drafts` → `posts` to respect FK constraints.

### Decision 4: Threshold default 0.55 Jaccard
**Chosen**: 0.55 (lower than the humanizer's 0.70 threshold).  
**Rationale**: Cross-corpus pairs above 0.55 are unlikely to be coincidental on domain-specific Indian tax content. The humanizer uses 0.70 to detect regressions on the *same* article; cross-article similarity at 0.55 already indicates substantial overlap. Configurable via `--threshold` flag.

## Risks / Trade-offs

- **Irreversible deletion** → Mitigation: scripts default to `--dry-run`; actual deletion requires `--confirm` flag. Git history preserves deleted files.
- **O(n²) comparisons slow on large corpora** → Mitigation: acceptable for ≤1000 posts; add a `--limit` flag to cap comparison count if needed.
- **Bigram-Jaccard misses paraphrased duplicates** → Mitigation: flag is only advisory; human reviews the report before deleting.
- **D1 FK constraints (generation_jobs → posts)** → Mitigation: deletion script deletes in correct dependency order: `generation_jobs` first, then `drafts`, then `posts`.
- **drafts table FK**: The existing `d1-schema` spec doesn't define FK for `drafts → posts`. The deletion script will query by `post_slug` column (or join) rather than relying on FK cascade.

## Migration Plan

1. Run `npx tsx scripts/detect-similar-posts.ts` → review `similar-posts-report.md`.
2. Identify slugs to delete from the report.
3. Run `npx tsx scripts/delete-similar-posts.ts --slugs <slug1,slug2> --dry-run` → verify output.
4. Run with `--confirm` to execute deletion.
5. Rebuild Astro site to verify no broken references.

No rollback beyond Git (`git checkout src/content/blog/<slug>.md`); D1 rows cannot be restored without a backup.

## Open Questions

- Should deleted slugs be added to a denylist to prevent future topic-discovery from re-generating them?
